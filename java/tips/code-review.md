## 基本のレビュー観点
### 🧩 コードの正確性
- ロジックは仕様通りに動作しているか
- 不要な条件分岐や重複コードはないか
- nullチェック・例外処理は適切か

### 🧩 可読性・命名
- クラス名・メソッド名・変数名は意味が伝わるか
- 不要な略語や曖昧な命名を避けているか
- コメントが過剰・不足どちらにも偏っていないか

### 🧩 コーディングスタイル
- チームの コーディング規約（命名規則・インデント・括弧の位置など） に沿っているか
- 定数はstatic finalで管理されているか
- マジックナンバーが埋め込まれていないか

### 🧩 再利用性・拡張性
- 同じ処理が複数箇所に書かれていないか
- 将来的な拡張を考慮した設計になっているか
- インターフェースや抽象クラスを使って依存を減らせるか

### 🧩 パフォーマンス・効率性
- 不要なオブジェクト生成やループがないか
- コレクション操作（List, Map, Stream）は適切か
- スレッド処理・同期制御は安全に行われているか

### 🧩 セキュリティ
- 外部入力（フォーム、API、SQLなど）に対してバリデーションしているか
- パスワードや機密情報がログや例外に出力されていないか
- SQLインジェクション・XSS対策が取られているか

## 🧩 Java特有のチェックポイント
| 項目 | チェック内容 |
|-----|-----|
|例外処理|try-catch の粒度が適切か？ throwsを乱用していないか？|
|コレクション|List, Set, Map の選定が適切か？|
|Stream API|可読性を損なわない範囲で使われているか？|
|オブジェクト比較|== ではなく .equals() を使っているか？|
|リソース管理|try-with-resources 構文を活用しているか？|
|スレッド処理|ExecutorService や CompletableFuture を適切に使っているか？|
|ログ出力|System.out.println ではなくロガーを使用しているか？|

## 🧠 レビュー時の心構え
| 項目 | 内容 |
|-----|-----|
|🤝 攻撃的ではなく建設的に|「何がダメか」ではなく「どう良くできるか」を伝える|
|💬 背景を尋ねる|なぜこの実装にしたのかを質問することで理解を深める|
|📚 コード規約を共有|チーム全体で統一ルールを明文化しておく|
|🕒 時間をかけすぎない|レビューは完璧を目指す場ではなく、リスク軽減の場|
|🌱 学びを共有|よい実装例は「ナレッジ化」してチームに残す|


## 🧩 レビュー指摘の一例
| Before | After | 改善理由 |
|-----|-----|-----|
if (user != null && user.getName() != null) | Optional.ofNullable(user).map(User::getName).orElse("") | Nullチェックをシンプルに
System.out.println("Error!") | logger.error("Error occurred: {}", e.getMessage()) | ロガーを使って情報を記録
List list = new ArrayList(); | List<String> list = new ArrayList<>(); | ジェネリクスで型安全に
if (flag == true) | if (flag) | 冗長な条件を簡潔に
